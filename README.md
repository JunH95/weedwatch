# weedwatch

주말농장을 대신 돌보는 **자율 제초 로봇**을 시뮬레이션 안에서 만든다.
카메라로 작물과 잡초를 구별해 잡초 위에 정밀하게 서서 제거한다.

> **현재 상태: Stage 0/7 완료.** 아직 로봇도 정원도 없다.
> 지금 있는 건 "화면 없이 GPU 시뮬을 돌려 사진을 받아올 수 있다"는 증명 하나다.
> 진행 상황은 [STATUS.md](STATUS.md), 설계 결정은 [docs/DECISIONS.md](docs/DECISIONS.md).

## 왜 이 프로젝트인가

**기존 제초 로봇은 전부 "보는 것"을 우회한다.**

| | 어떻게 잡초를 고르나 | 카메라로 보나 |
|---|---|---|
| **Tertill** (유일했던 가정용 제초 로봇, 단종) | 키 규칙 — 2.54cm보다 크면 작물, 작으면 잡초 | ❌ 용량성 센서 |
| **FarmBot** (텃밭 위 고정 갠트리) | 자기가 심어서 좌표를 안다 | ⚠️ 있지만 판단은 좌표 DB가 |
| **상업용 제초 로봇** | 줄 기하학 — 줄 위는 작물, 사이는 잡초 | ⚠️ 줄을 따라가는 용도 |
| **농진청 과수원 제초로봇** | 안 고른다. RTK로 다니며 하부 회전날로 전부 예초 | ❌ 장애물 감지 전용 |
| **핑거위더·토션위더 등 기계 제초 도구 전체** | 작물과 잡초의 **크기 차이** | ❌ |

마지막 줄이 핵심이다. **핑거위더는 기계로 만든 키 규칙이다.**
즉 이건 Tertill 하나의 선택이 아니라 **기계 제초라는 분야 전체가 크기 차이로 선택성을 사는 것**이고,
카메라는 그 의존을 끊는 유일한 방법이다.

### 왜 하필 주말농장인가 — 규정이 문제를 만든다

상업 밭에는 제초 문제가 없다. **비닐 멀칭과 제초제가 있으니까.**
그런데 지자체 주말농장은 그 둘을 **규정으로 금지한다**:

> "유기합성농약, 화학비료를 사용하지 않는 유기농법과 **비닐 멀칭을 하지 않는** 환경친화적인 농법을 준수"
> — 서울시 동행서울 친환경농장 (남양주·경기도 동일)

**제초 로봇은 비닐+제초제의 여집합이다.** 이 세그먼트는 두 산업적 정답의 부재로 정의된다.
(규모: 도시농업 참여자 150.4만명 · 도시텃밭 952ha — 농식품부 2024)

### 프로젝트 질문

**눈을 달면 Tertill의 제약이 몇 개나 사라지나?** 그리고 정직한 예상 답은 "전부"가 아니다:

- **사라진다**: 울타리 필수 · 잡초 없는 상태에서 시작 · 랜덤워크 · 20㎡ 한계
- **안 사라진다**: 25~30cm 작물 간격 — 이건 비전이 아니라 **차체 폭(20.96cm)**의 산물이다

측정 결과가 전부 초록불인 표는 아무도 믿지 않는다. 그래서 지는 칸을 일부러 찾는다.

### 로봇은 큰 잡초를 못 죽인다. 그래서 사람에게 넘긴다.

제초 기술 전체가 어린 잡초에만 듣는다 — FarmBot 공식 문서조차 *"Do not expect this tool,
or FarmBot in general, to be able to remove large, established weeds"* 라고 쓴다.
레이저는 치사량이 떡잎→4엽에서 약 1000배 오른다. **큰 잡초용 제초 기술은 없다.
일찍 제초하는 것만 있다.**

그래서 로봇은 잡초를 **3분류**한다:

| 결과 | 동작 |
|---|---|
| 자율 처리 | 스탬핑 |
| **사람에게 넘김** | 위치와 사진을 남긴다. 주말에 5분이면 뽑는다 |
| 놓침 | 탐지 실패 ← **유일한 진짜 실패** |

**여기가 이 프로젝트의 핵심이다: Tertill은 가운데 칸을 구조적으로 못 채운다.**
큰 잡초는 Tertill에게 *작물*이다. 못 죽이는 게 아니라 **알지도 못한다.**
키 규칙의 실패 방식이 조용한 오분류라서, *"내가 못 다루는 걸 발견했다"* 를 표현할
언어 자체가 없다. **비전이 주는 건 제거 능력이 아니라 분류 능력이다.**

그리고 이게 "주말농장"이라는 이름과 맞아떨어진다 — 로봇은 **항상 있지만 약하고**,
사람은 **강하지만 주말에만 있다.** 로봇이 시간이 중요한 약한 일을 하고, 사람이 드물어도
되는 강한 일을 한다. 대시보드는 **시정수가 다른 두 주체를 잇는 인터페이스**다.

## 검증 방식

이 저장소의 특이한 제약: **사람이 GUI를 안 본다.** AI 에이전트가 만들고 스스로 검증한다.
따라서 **CLI에서 단언할 수 없는 기능은 존재할 수 없다.**

```bash
make doctor   # 환경 건강검진 (파이썬 3.10 / rclpy / EGL / NVIDIA)
make smoke    # 화면 없이 GPU 렌더링 → 사진 → 게이트 2개 단언
```

**모든 시뮬 테스트는 게이트가 2개다**: 사진이 나왔나 **AND** NVIDIA가 그렸나.
사진만 보면 거짓 통과한다 — 이 컴퓨터의 EGL은 기본값으로 두면 RTX 4060을 못 찾고
인텔 내장 그래픽을 잡는데, 그림은 *멀쩡해 보이면서* 100배 느리다.

```
$ make smoke
게이트 1 — 그림
  사진 장수   : 121
  평균/표준편차: 142.00 / 36.12
게이트 2 — 장치
  GL_VENDOR   : NVIDIA Corporation
  GL_RENDERER : NVIDIA GeForce RTX 4060 Laptop GPU/PCIe/SSE2

통과: 화면 없이 GPU 렌더링이 확인되었습니다.
```

## 기술 스택

| | |
|---|---|
| 시뮬 | Ignition **Fortress** 6.17.0 + ROS 2 **Humble** (Ubuntu 22.04) |
| 정원 생성 | [CropCraft](https://github.com/Romea/cropcraft) (INRAE) + Blender 5.0.1 |
| 인식 | 의미분할 (합성 데이터로 학습) |
| 로봇 | 포탈형 — 고랑에 바퀴, 두둑을 타고 넘음. Y/Z 직교 관절로 잡초에 접근 |
| 제거 | **스탬핑** — Bosch BoniRob 방식(1cm 막대로 흙 속 3cm 압입) |

**제거를 자르기가 아니라 스탬핑으로 하는 이유**: 스탬핑은 강체 물리라 정직하게 시뮬된다.
자르기는 연성체 물리가 필요한데 Gazebo에 없다(`gz-physics#222`, 2021년부터 미해결).
그리고 더 중요한 이유 — Tertill이 증명했듯 **제거에는 지능이 필요 없다. 어느 놈을
제거할지 고르는 데 필요하다.** 효능이 아니라 선택성이 이 프로젝트의 기여다.

### 이 시뮬이 보여줄 수 없는 것 (먼저 밝힌다)

- **"눌린 잡초가 죽었다"는 못 보여준다.** 연성체 물리가 없어서 스탬핑을 골랐는데,
  같은 이유로 치사 여부도 시뮬할 수 없다. 시뮬이 보여주는 건 **"올바르게 분류하고
  둘 다 실행했다"**이지 **"죽었다"**가 아니다.
- **"몇 cm부터 못 죽이나"는 측정이 아니라 인용이다.** 에스컬레이션 임계값은
  문헌에서 고정하고 튜닝하지 않는다.
- **인식 성능을 시뮬 숫자로 주장하지 않는다.** 같은 렌더러로 학습하고 평가하면
  거의 암기 과제다. 그래서 AI Hub 실제 잡초 사진(쇠비름 — CropCraft 에셋과 같은 종)에
  돌려서 나오는 숫자를 정직하게 보고한다.
- **발표된 로봇 성능과 나란히 비교하지 않는다.** 재는 물리량이 다르다 —
  우리는 "잡초 위에 섰나", 저쪽은 "잡초가 죽었나"(밀도/피복률/생체량).
  단위 변환이 존재하지 않는다.

*Isaac Sim은 기각: 공식 최소 사양이 VRAM 16GB인데 이 노트북은 8GB. 그리고 필요도 없다 —
학습 이미지는 Blender에서 오프라인으로 렌더링하므로 사진 품질이 런타임 시뮬레이터와
분리된다. 어느 정밀도가 실제로 중요한지의 문제다.*
