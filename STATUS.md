## Stage 1-a — 두둑/고랑 기하학 (완료, 위험 제거됨)

**이번 단계 최대 위험이었다**: CropCraft에 지형 개념이 아예 없어서(바닥이 평면)
두둑을 직접 만들어야 하는데, 포탈형 설계 전체가 두둑 높이 25cm에 걸려 있었다.

**가장 싸게 죽였다**: 위험의 본질은 "로봇이 탈 수 있는 기하학인가"이지
"CropCraft와 통합되나"가 아니다. 그래서 **Blender 없이 상자 프리미티브만으로** 만들었다.
반나절 예상 → 30분.

**결과: 된다.** `make test` 21개 통과 · `make garden` 렌더 확인.

```
두둑 2개 · 폭 90 · 높이 25 · 고랑 30cm
→ 로봇 트랙 120cm · 전폭 128cm · 바퀴 좌우 여유 11.0cm/쪽
```

**여기서 배운 것 — 눈으로 본 게 버그를 잡았다.** 첫 렌더에서 로봇이 두둑이 아니라
**고랑을 걸터타고 있었다** — 바퀴가 두둑 위에 올라가고 몸통이 고랑 위에 뜬, 설계와
정확히 반대인 모습이었다. 두둑이 짝수 개면 y=0이 고랑 한가운데인데 로봇을 거기 세워서다.
숫자(트랙 120, 여유 11cm)는 전부 "통과"했고 사진을 보고서야 알았다.
→ 그래서 `tests/test_garden_geometry.py`가 이제 산수로 검사한다. 눈은 확장이 안 된다.

**같이 확정된 것**: v1 작물은 **상추만**. 배추(40cm)는 75cm 포탈이면 들어오지만 안 올린다 —
CropCraft가 작물로 콩/옥수수만 배포해서 상추도 배추도 결국 같은 콩 메시를 크기만 바꾼 것이
되고, 그러면 둘을 가르는 신호가 크기뿐이라 "비전이 키 규칙을 이긴다"가 순환논법이 된다.
형태 다양성은 잡초 쪽(portulaca/polygonum/taraxacum — 서로 다른 메시)에서 온다.

**남은 v1 단순화** (나중에 갚을 빚): 두둑이 옆면 수직인 상자다. 진짜 두둑은 비스듬하다.
"예쁜가"가 아니라 "탈 수 있나"를 물었으므로 지금은 충분하다.

## Stage 1-b — Blender + Cycles GPU (완료)

Blender **5.0.1** (snap `5.0/stable` 고정, 번들 파이썬 3.11.13).

**함정 확인 + 해결**: 새로 깐 Blender는 `compute_device_type = 'NONE'`이라
Cycles가 **경고 한 줄 없이 CPU로 렌더링**한다. 스크립트에서 `device='GPU'`라고 써도
조용히 무시된다. 환경변수 우회 없음 — 설정 파일을 만드는 게 유일한 방법.
→ `make blender-gpu` (OPTIX로 켜고 `userpref.blend` 저장). `make doctor`가 매번 검사한다.

**그 검사가 진짜 실패하는지도 확인했다**: 설정 파일을 치우면 `exit=2`로 죽으면서
원인과 고치는 법을 출력한다. 통과만 확인하면 절반이다.

**측정된 렌더 시간** (1920×1080 / 32샘플 / 식물 800개 / OPTIX):

```
장면 빌드   8.06s   (여러 장에 분산됨)
장당 렌더  11.2s   (11.60 → 11.14 로 수렴)
```

→ **논문 헤드라인 재현(1,050장) ≈ 3.5시간.** 하룻밤이면 된다.
리서치 추정(5~10초/장)보다 느리지만 결론은 유지 — 일주일이 아니다.
그리고 논문 헤드라인이 1만 장이 아니라 **1,050장**이었다는 게 여기서 값을 한다.
(공개된 1만 장 데이터셋 `doi:10.57745/TNCSLP` 도 있어서 첫 베이스라인엔 렌더링이
아예 불필요할 수도 있다.)

### diff-drive 를 붙이며 밟은 함정 3개 (전부 검증에서 잡힘)

주행이 처음엔 안 됐다. odom 은 "전진 1.9m" 라고 보고하는데 몸통은 0.001m 도 안 갔다 —
**게이트를 둘로 둔 이유가 정확히 이거다.** odom(플러그인)만 봤으면 거짓 통과했다.

1. **관성 `izz`가 0 으로 반올림돼 DART 가 물리를 통째로 깨뜨렸다.** tool(가는 막대)의
   izz=3.6e-6 을 `%.5f` 로 찍으니 `0.00000`. DART 가 무효 관성으로 보고 관절체 알고리즘이
   망가져 바퀴가 떴다. → `%.8g` + "관성 대각성분>0" 가드. (`tools/inertia.py` 산수 검증)
2. **빔/캐리지가 바퀴 아래로 매달려 로봇이 캐리지로 서고 바퀴가 5mm 떴다.** `beam_z` 가
   clearance(0.60)가 아니라 pod 아래 0.22 에 있어 캐리지·카메라·도구가 지면 근처로 내려왔다.
   → `beam_z = clearance + beam_height/2` 로 수정(빔 = 터널 천장). 재-export.
   이게 아래 "디자인 미결"의 **"도구가 땅에 닿아 보임"** 을 실제로 해결했다.
3. **tool 충돌/관성 길이가 행정(z_travel=0.35)이라 시각 막대(0.1925)보다 길었다** — 접힘
   자세에서 두둑을 파고든다(평지 월드라 잠복). → `Portal.tool_rod_len` 단일 출처화. (적대적
   검증에서 잡힘.) 나머지 확인: 좌/우 배정·wheel_separation·단언 이중게이트는 검증 통과.

### 디자인 미결 (사용자가 "일단 놔두고 진도" 결정 — 나중에 돌아옴)

지금 로봇은 "책상"은 벗어났지만 사용자가 아직 불만. **파라메트릭이라 나중에 고치기 쉽다**
(치수만 바꾸면 URDF·시뮬 그대로). 돌아올 때 고칠 것:
- 옆 포드 ↔ 상판 데크 **연결이 부자연스러움** (처마가 붕 뜬 느낌)
- **바퀴 ↔ 옆 포드 연결 어색** (다리가 애매하게 보임)
- ~~도구 막대가 땅에 닿아 보임~~ → **해결** (빔을 clearance 높이로 올려 도구가 터널 안으로).
- **색이 흰-회색뿐** — 초록 악센트/투톤(AVO는 흰+초록) 넣으면 "제품" 느낌
- 사이드 포드 옆 홈이 너무 큼
- **카메라 높이가 스펙의 절반** (적대적 검증 발견) — 카메라가 두둑 위 0.18m 에 있는데
  `camera_height_above_bed=0.35`(px/cm 계산 근거)의 절반이고 작물 캐노피(0.50) 아래로
  잠긴다. 빔 수정으로 0.01→0.43 개선됐지만 스펙(0.60)엔 못 미침. **인식(Stage 3) 전에
  해결 필요** — 스펙 값을 파생 속성으로 만들어 두 곳에 안 적히게. (지금은 카메라 센서 링크가
  없어 잠복 — px/cm 을 읽는 코드도 없음.)
- **배터리 시각 메시가 터널에 떠 있음** (적대적 검증 발견) — 포드 안이 아니라 가운데 공중.
  시각 전용(충돌 없음)이라 물리 영향 0, 외관만.
- **캐리지 충돌 상자가 시각 본체보다 8.5cm 낮음** (적대적 검증 발견) — 링크 원점이
  카메라·LED 포함 그룹 bbox 중심이라. 바퀴 접지면 위라 주행엔 무관, 미미.

## Stage 5 — 지형·흔들림 대응 자율 타격 (착수, `docs/DECISIONS.md` 025)

논지(사용자): 밭을 물리적으로 울퉁불퉁하게 만들어 로봇을 실제로 기울고 흔들리게 하면 스탬핑 계산이
달라진다 — 이걸 IMU·깊이로 맞춘다. 실물 로봇 없이도 프로젝트 격을 올리는 지점. Tier 1 준정적 기울기
→ Tier 2 동적 요철, 센서 실측 노이즈, 보정 A/B 단언.

### Tier 1 선검증 — 기울기 + IMU 기반 (완료, `make tilt`)

착수 전 확인한 사실:
- **어떤 월드에도 imu-system 플러그인이 없었다** → URDF 에 imu 센서가 있어도 `/robot/imu` 가 발행조차
  안 됐다. IMU 는 "발행만 되는 유휴"가 아니라 아예 죽어 있었다.
- 로봇은 평지(furrow_floor z=0 평면 + 두둑 상자 충돌)를 달려 **절대 안 기운다** — 사용자 지적대로.

한 것: `tools/make_tilt_world.py`(크로스슬로프 roll=5° 지면 + imu-system, 단일 출처 TILT_DEG) +
`tools/assert_tilt.py`(게이트 2개). 렌더 없음 → Tier 2 물리 시뮬.
- **게이트 B(물리)**: GT roll = +5.00°(부여값 정확), pitch~0, z 안정, roll std 0.00° → DART 가 기운
  접촉에서 안 터지고 발산 없음. 상자를 쓴 이유가 DART 안정이었는데, 사면/기울기도 안정임을 확인.
- **게이트 A(센서)**: IMU roll = +5.00°, GT 와 Δ0.00° → IMU 발행·정확.

**정직한 발견**: gz IMU 의 orientation 은 노이즈 없는 지상진실 자세에서 계산돼 GT 와 거의 같다(적분 안
함). 실물 IMU 자세추정은 드리프트가 있다 → 이 갭은 Stage 5 후반 orientation 노이즈로 메운다(원장 추가
예정). 지금 게이트(배선+DART)엔 무해.

### Tier 1 A/B — 기울기 보정 스탬핑 (완료, `make tilt-stamp`)

`tools/make_tilt_world.py stamp`(8° 크로스슬로프 고랑 + 수평 두둑 + imu-system) +
`tools/assert_tilt_stamp.py`(무보정 vs IMU 보정 두 솔버, 전체회전 FK 채점). 잡초 3개(밴드별 1개):
- **무보정(수평 가정)**: 도구가 기운 몸통 -z 로 하강 → tanφ·깊이만큼 밀려 **3.57~4.26cm 빗나감(전부 MISS)**.
- **IMU 보정**: roll 로 옆밀림 상쇄 → **0.05~0.10cm(전부 HIT)**. 제어=IMU(실물 잔차 bias 0.8°/노이즈 0.5°
  얹음, 전처리 안 함), 채점=지상진실. **3회 반복 동일 — flaky 아님.**

정직 처리(사용자 Q): 센서 노이즈가 순환 아니냐 → 아니다. 전처리는 랜덤 실현값을 못 되뺀다, 잔차가
남아 성능을 깎고 그걸 잰다. 순환은 "센서가 몰래 GT일 때"뿐인데 그게 gz IMU orientation(오차 0)이라,
실물 BNO085 동적 잔차 ~1°를 얹어 그대로 쓴다. (DECISIONS 025 보정.)

### Tier 2 Step A — 동적 요철 주행 소양 (완료, `make shake`)

`tools/make_shake_world.py`(고랑 바퀴 경로에 시드 고정 흙덩이 17개, 높이 3~6cm) + `assert_shake.py`
(게이트 3개). 로봇이 요철 밭을 주행하며:
- **완주/안정**: 전진 2.46m, 최대기울기 3.0°, z 안정, NaN 없음 → **DART 가 범프에서 안 터진다**
  (상자 충돌을 쓴 이유가 안정성이었는데, 절차적 범프도 안정). 3회 반복 동일.
- **흔들림**: roll p2p 4.1° / pitch p2p 5.8° → 실제로 시변(보정할 게 생김).
- **IMU 추적**: 시변 자세를 |IMU−GT| 중앙값 0.01° 로 따라옴 → 주행 중 보정 입력으로 쓸 수 있다.

Step B 함의: gz IMU 는 동적으로도 GT 를 0.01° 로 추적(참자세 계산). 그러니 Step B 의 타이밍 난제는
IMU 지연이 아니라 **하강 180ms 동안 몸이 더 기우는 것** — 타격 순간이 아니라 착지 순간 자세를
예측해야 하는가를 잰다(shake ~1Hz 라 180ms 변화가 얼마나 되는지가 관건).

**Step B 계획**: 요철 위 주행하며 타격 A/B(무보정 vs IMU-타이밍 보정) + 착지시각 예측 필요성 측정.
측정 결과 하강 180ms 동안 자세 변화는 중앙 0.5°/최대 1.9° → 타격오차 최대 ~0.8cm(2차 항)이고,
범프만으로는 무보정 오차도 ~1.2cm 라 2cm 안에 들어 A/B 가 안 갈린다 → **Step B 지형은 크로스슬로프
+ 흙덩이를 합쳐야** 한다(무보정 ~4.7cm FAIL / 보정 PASS).

## 카메라 2대 — 한 대로는 두둑을 못 덮었다 (`docs/DECISIONS.md` 026)

사용자 질문("카메라가 수평으로 달렸나, 앞을 보게 기울이면?")을 확인하다 **더 큰 문제**를 찾았다.
D405 한 대는 두둑 위 0.33m 에서 가로 **0.585m** 밖에 못 보는데 툴은 **0.90m**(±0.45)까지 닿는다 →
**35%가 사각**. 주행은 x 축이라 이 y 공백을 못 메운다(로봇 y 는 고랑에 고정). ⟹ **완벽한 검출기라도
재현율 상한 0.65** — 관측된 0.25~0.5 의 상당 부분이 이것이었다. 코드 주석("두둑 전체를 봄")과
DESIGN §9("0.35m면 90cm 커버")가 **산수를 안 해본 틀린 주장**이었다(둘 다 정정함).

- **해결**: 카메라 **2대**(2× D405), y=±bed_width/4=±0.225. 커버 반폭 0.5175 ≥ 툴 도달 0.45,
  가운데 겹침 0.135m(사각 없음). 앞으로 기울이기는 기각(리드타임은 이미 8~19배 남고, off-nadir 은
  시차·가림·흔들림 민감도를 전부 악화).
- **단언**: `tests/test_camera_coverage.py` 7개 — **산수는 산수로**. 시뮬로는 절대 안 잡힌다
  ("안 보이는 잡초"는 시뮬에서도 안 보여 통과처럼 보인다).
- **회귀**: 단위 64개 · straddle · stamp(6/6 ≤0.02cm) · tilt-stamp 전부 통과.
- **융합 완료**: detect_server 가 카메라별 y 오프셋으로 픽셀→world + 겹침 dedup(`merge_detections`,
  Tier 1 8개). `make percept` 2회 동일: 시야 안 target **2개 → 4개**, 검출률 4/4=1.00.
- **프레임률**: 2대로 늘리자 sim 이 0.200x → **0.031x**(6.5배 느려짐, GPU 경합) → row-live 완주 실패.
  프레임률은 정밀도가 아니라 **비용 노브**라 15Hz→**5Hz** 로 낮춤(`camera_rate` 단일 출처) →
  **0.344x**. 카메라 2배인데 예전 1대보다 1.7배 빠름. 하한은 `tests/test_camera_rate.py` 가 단언
  (프레임 간격 4cm vs 주행방향 발자국 32.9cm, 잡초당 8.2장).

### 정정 — 이 밭에서는 커버리지가 병목이 아니었다

2대로 row-live 를 완주시켰더니 재현율이 **0.25 그대로**. 재보니 **robot_field 주행 구간 잡초 15개가
전부 중심 ±0.225m 안**(한 대 반폭 0.292 안) → **2대가 새로 본 잡초 0개.** 산수(한 대가 툴 도달폭의
65%만 봄 = 구조적 상한 0.65)는 여전히 참이고 전폭에 잡초가 퍼진 밭에서는 실제 병목이지만, **지금
쓰는 밭은 두둑 폭 절반에만 잡초를 심어서** 그 상한에 안 닿는다.

⟹ **남은 병목은 검출→타격 사슬.** 실측: 검출 13개 vs 표적 12개로 검출은 충분한데 8cm 안 타격은 3개,
놓친 표적은 도구가 9~42cm 떨어져 지나갔다(도구를 갖다 대지도 못함). 후보: 늦은 공개(카메라 리드 대비
strike_x), safe-dist 제외, 툴 점유 충돌, 검출 y 오차의 밴드 오배정. **다음 작업 = 이 사슬 계측.**

## 2026-07-20 — 사용자 대화로 계획 정련 (`docs/DECISIONS.md` 013)

로봇 몸체(주행·관절·걸터타기·카메라)를 다 검증한 뒤, 사용자 질문들로 설계가 여러 개 정련됨:
- **작물 = 콩 + 옥수수** (가짜 상추 폐기 — 둘 다 실제 한국 텃밭 작물, 서로 다른 메시).
- **straddle = 낮은 작물 제초기.** 클리어런스(빔 60 − 두둑 25 = **작물 최대 35cm**)가 서비스
  범위. 콩은 철 내내, 옥수수는 넘기 전(~medium)까지. **"작물이 클리어런스 넘음" = 대시보드
  보고 상태** (잡초 triage 와 대칭 — 도달 범위 넘는 건 조용히 무시 말고 보고).
- **제거 = 사라짐 처리**(죽음 시뮬 안 함, 002). **추가**: 도구가 필요 깊이(문헌 3cm) 도달은 단언.
- **stamp vs hoe 정정**: 둘 다 bookkeeping 이라 구현 가능. stamp 는 **선택성** 때문에 고른 것
  (물리 장벽 아님). "자르기=연성체" 는 실제 절단 물리 얘기였음.
- **카메라 높이 = 기능**(서비스 작물키 결정) → 빔 높이로 올려야. 앞으로 뺀 팔에.
- **키 규칙 비교 = 곁다리로 강등.** 헤드라인은 우리 비전의 선택 정확도. "최신 대비"는 P4 근접도.
- 시나리오(정원·잡초분포·커버리지)는 **Stage 4~5 에서 설계** (그대로).
- **실물 제작 가능 설계 종합 → `docs/DESIGN.md`** (사용자 요구: "실제 로봇·서비스 만들 수준").
  6 서브시스템 병렬 설계: 실 부품 BOM ~$2,900(D405·Jetson Orin Nano·ZLTECH 인휠·LiFePO4·
  FastAPI+RN 앱) + **심-리얼 갭 원장** + **확정 필요 교차결정 9개**(4클래스·카메라높이·12V/24V·
  실물질량~40kg·D405 intrinsic 재보정 등). AI Hub 쇠비름 실사진이 **흰 배경**임을 눈으로 확인 →
  sim-to-real 갭이 예상보다 큼(원장에 기록).
- **로봇 모델을 buildable 수준으로 정련 (Stage 3 전 완결)**: 카메라를 전방 팔에 올려 두둑 위
  ~0.33m 에서 봄(RealSense D405, 1280×720/87°, LED 링), 배터리·전장함을 터널→**포드 안 낮게**
  (뜬 부품 없음, COM z 0.51→0.44 낮아짐), Y/Z 스텝모터·리드스크류 시각화(구동이 보임),
  실물 질량 ~34.5kg 반영(허브모터 2.5kg×4·배터리·전장). 도구 게인 재튜닝(P 2000). **drive/

## 깊이 융합 착수 — 원본 깊이를 손에 넣었다 (Stage 5)

인식 오차 3~5cm(잎 중심 vs 밑동 + 시차)를 줄이려면 깊이가 필요한데, 받는 경로부터 막혀 있었다.

- **깊이 `<save>` 는 쓸 수 없다(실측)**: 태그가 먹기는 해서 PNG 가 떨어지지만 **8비트 RGB 정규화
  시각화**다(고유값 192개). 3~5cm 를 고치는 데 못 쓴다. → 되돌림(I/O 비용만 듦).
- **`build/ww_depth`(신규)**: `ww_cmd` 와 같은 이유로 ign-transport 직결 C++ 상주 구독자.
  `/robot/depth{,1}`(ignition.msgs.Image, R_FLOAT32, 미터)를 받아 `artifacts/depth{i}/{seq}.bin`
  으로 내린다(형식 `[u32 w][u32 h][f32...]`, 원자적 rename). 파일명 seq 로 RGB 프레임과 정합
  (시각 정합은 5Hz 에서 최대 200ms=4cm 어긋나 우리 오차와 같은 크기라 못 씀).
- **검증**: 카메라 2대 × 68프레임 수신. 값이 기하와 일치 — 고랑 바닥 0.591m(기대 0.58), 두둑
  표면 ~0.33m, 콩 캐노피 꼭대기 0.131m(두둑 0.25+작물 0.22=0.47 → 0.11). 즉
  **표면 z = camera_z − depth** 로 식물 키가 직접 나온다.

1. **카메라 커버리지가 재현율 병목이라 했는데** — 이 밭 잡초 15개가 전부 중심 ±0.225m 안이라

   (scale 1.00). 보정하니 오히려 4.51→5.63cm 악화. **기각**(028). 깊이는 정확했다(콩 22.9cm).

## 막힌 곳 (오늘 발견한 진짜 병목)

**경사 6° + 흙덩이에서 휠 오도메트리가 무너진다** — GT 실제 이동 0.844m vs odom 보고 2.6m,

오차(44~181cm)가 된다. **base 가 1m 틀렸는데 캐리지를 cm 로 보정하는 건 무의미하다.**
