# weedwatch — 구현 계획

## Context

노션 「주말농장 관리 로봇 (weedwatch)」을 실행 가능한 계획으로 구체화한다. 현재 코드 0줄, `/home/sunny/projects/weedwatch`는 비어 있고 git 저장소도 아니다.

**논지**: 기존 해법은 전부 비전을 우회한다 — Tertill(키 규칙+랜덤워크), FarmBot(심은 좌표 DB), 상업용(줄 기하학), **농진청 과수원 제초로봇**(RTK 경로추종 + 하부 회전날 무차별 예초, 카메라는 장애물 전용), 그리고 **기계 제초 정전 전체**(핑거·토션·바스켓 위더가 전부 크기 차이로 선택). 취미 텃밭은 이 편법들이 통하지 않는다.

**대상 세그먼트 (규정으로 정의됨)**: **무멀칭 친환경 주말농장.** 서울시 동행서울 친환경농장 규정 원문 — *"비닐 멀칭을 하지 않는 환경친화적인 농법을 준수"*(남양주·경기도 동일). **상업 밭엔 제초 문제가 없다 — 비닐과 제초제가 있으니까. 주말농장에 제초 문제가 있는 건 그 둘이 금지돼서다.** 제초 로봇은 비닐+제초제의 여집합이다. 규모: 도시농업 참여자 150.4만명 · 도시텃밭 952ha(농식품부 2024).

**완성 기준 (kill event)**: 절차 생성된 정원에서 자율 주행 → 카메라로 잡초 식별 → 그 위에 정밀 위치 → **스탬핑**(BoniRob 방식, 강체 물리라 정직하게 시뮬됨) → **작물은 안 건드림**. 자르기는 범위 밖(연성체 필요, Fortress에 없음 — 검증됨).

**노션에 없던 새 제약**: *Claude가 만들고 Claude가 스스로 검증한다.* 이것이 아키텍처를 결정한다 — GUI 없이 CLI에서 단언 가능해야 기능이 존재할 수 있다.

**사용자 결정**: 순서는 Claude가 정함 / 스코프는 결국 전부 / 실물 촬영 없음(한국 공개 데이터셋 우선) / 로봇 형태는 도면 보고 결정 → [게시된 도면](https://claude.ai/code/artifact/64274c0f-9218-4980-9f3c-f72ec916b5f6) / 베이스라인 비교 원함.

### 🔴 이 머신은 이미 잘못된 GPU로 렌더링 중

`~/.ignition/rendering/ogre2.log` 실제 기록:
```
Found Num EGL Devices: 2
EGL Device: #0 /dev/dri/card1               ← Intel UHD
EGL Device: #1 EGL_MESA_device_software     ← llvmpipe
GL_RENDERER = Mesa Intel(R) UHD Graphics (ADL-S GT1)
```
**RTX 4060이 EGL 목록에 없다.** gz-sim#1272(미해결)로 항상 index 0이 선택되고, gz-sim#1116에 따라 llvmpipe는 **검은 화면**을 낸다.

- 고정: `__EGL_VENDOR_LIBRARY_FILENAMES=/usr/share/glvnd/egl_vendor.d/10_nvidia.json` — 모든 실행 경로에.
- `/dev/nvidia*`는 0666 → X 없이 접근 가능. `usermod -aG render` **불필요**(renderD128은 인텔, NVIDIA EGL은 `/dev/dri` 미사용).
- **모든 시뮬 테스트는 게이트 2개**: 픽셀(검지 않음) **AND** 디바이스(`GL_VENDOR = NVIDIA`). 픽셀만 보면 거짓 통과 — llvmpipe는 *올바른* 이미지를 100배 느리게 낸다.
- `-r` 필수: 시뮬은 기본 일시정지. 없으면 클럭이 안 돌아 카메라가 안 뜨고, EGL 고장처럼 보인다.

### Python — 지금 깨져 있음

`python3` → miniforge **3.13.13**이 `/usr/bin/python3`(3.10.12)를 가림 → `import rclpy` 실패. `~/.bashrc`가 4개 워크스페이스를 PYTHONPATH에 주입(비대화형 가드가 있지만 에이전트 셸은 프로필에서 초기화되어 무력).
→ `scripts/env.sh`가 **유일한 진입점**. ML은 `/usr/bin/python3 -m venv --system-site-packages`로 분리(conda 아님). ROS↔ML 계약은 디스크 파일(`models/best.pt`)이지 공유 import가 아니다.

### CropCraft — 쓰되 세 가지를 고쳐야 함

1. **Gazebo 출력이 Fortress에서 전부 검게 렌더됨.** Ogre1 `<material><script>`를 gz-sim이 무시하고 diffuse {0,0,0,1}로 메시 자체 머티리얼을 덮어씀. `<pbr><metal><albedo_map>` + `<double_sided>true` 변환 패스가 **모든 것의 선행 조건**.
2. **Cycles GPU가 GUI 클릭을 요구하고 없으면 조용히 CPU 폴백.** 1회성 `save_userpref` 스크립트 + `has_active_device()` 단언.
3. **메시 포맷은 OBJ 강제.** Fortress 로더는 DAE/OBJ/STL뿐(glTF 없음), PBR을 나르는 건 OBJ+MTL뿐, 그런데 **Blender 5.0이 DAE export를 제거**했다. 이 삼각형의 유일한 해가 OBJ.
4. **두둑/고랑이 없다.** CropCraft에 지형 파라미터가 아예 없어 바닥이 평면. 포탈 설계가 여기 걸려 있음 → per-bed `offset:[0,0,0.25]` + SDF에 정적 ridge 메시. **미검증, Stage 1에서 확인.**

### 🔑 잡초 정답 좌표 — 오라클 (성공 기준 측정의 전제)

CropCraft는 **작물 좌표만** export한다(`FieldState.beds[].rows[].crops[]`→`PlantState x,y,z`). 잡초는 geometry-node Poisson 인스턴스라 좌표가 Python에 존재한 적이 없다. trimesh로 OBJ를 쪼개는 우회는 **불가** — 측정 결과 `portulaca_01`은 72개 컴포넌트(잎 단위)로 쪼개짐.

**해결 (코드 0줄)**: crop/weed는 **에셋 속성이 아니라 설정에서 부여**된다(두 경로가 같은 `get_model_list_by_height()` 호출).
→ **채점 대상 잡초를 `beds:` 블록에 선언**. `description.json`에 정확한 x,y,z + 자체 SDF link + `laser_retro` ID가 붙는다. `weeds:` 블록은 채점 안 하는 방해물 전용.
→ 백업 경로: geometry-node의 Group Output을 realize 대신 raw points로 재배선하는 ~10줄 bpy 스크립트(검증됨). 두 경로가 일치하면 신뢰 가능한 오라클.

## 2. 논지 수정 — 노션의 전제가 반박됨

노션: *"취미 텃밭은 섞어심기에 줄도 고르지 않아서 이 편법이 전부 안 통한다."* → **틀렸고, 그대로 두면 리뷰어에게 반박당한다.**

- **Tertill은 줄과 무관.** 위치를 아예 모르는 랜덤워크다. 줄 유무와 직교한다.
- **실제 주말농장에는 두둑/고랑 구조가 있다**(농진청: 두둑 75~90, 고랑 30cm). "줄이 없다"는 사실과 다름.
- **정정: FarmBot은 카메라가 없는 게 아니다.** OpenCV HSV→blob→심은 좌표 DB 대조. "값싼 인식 + 강한 사전지식"이지 "인식 없음"이 아니다. 카메라 없이 구현하면 그게 허수아비다.

**교체할 논지 (더 강하고 정직함):**
- vs **Tertill**: 줄이 아니라 **키 분포의 겹침**이 키 규칙을 깬다. 반박 불가 실험 = 임계값 h 전 구간 스윕 → ROC. *어떤 임계값도 두 분포를 못 가른다*는 분포의 성질이라 조작 불가. **결정적 인용: Tertill 특허(US10888045)가 스스로 자백** — *"tall objects are considered crops, short objects are weeds"*이고, 모종은 잡초로 오인되니 전도성 보호대를 씌우라고 명시. 발명자가 문서화한 약점.
- vs **줄 기하학**: 줄이 있어도 안 통한다 — 잡초는 밭 전체에 뿌려져 **줄 위에도 얹힌다**. 줄은 선을 찾을 뿐 그 선 위 개체의 정체는 모른다.
- **가장 정직한 결론**: **Tertill의 25~30cm 간격 제약은 비전이 아니라 차체 폭(20.96cm)의 산물이라 눈을 달아도 안 사라진다.** 울타리·잡초 없는 시작·랜덤워크·20㎡ 한계는 사라진다. "전부 사라진다"보다 이게 날카롭다.
- **산수로 끝내기**: `Tertill 20.96cm > 상추 포기간격 20~30cm 중 하한` → 자기 목표 작물에서 물리적 통과 불가. 매뉴얼 요구 30.5cm를 전 범위에서 위반.

**리서치가 찾아낸, 더 강한 문장 3개:**
1. **논지는 Tertill보다 넓다.** 핑거위더·토션위더·바스켓위더 등 **기계 제초 정전 전체**가 *"size differential between crop and weed"*로 선택성을 산다 — **핑거위더는 기계로 만든 키 규칙이다.** 카메라는 그 의존을 끊는 유일한 것.
2. **키 규칙은 분류기가 아니라 복구 모드 없는 래칫이다.** Tertill의 2.54/3.81cm는 매일 순찰로 떡잎을 죽여야만 성립한다. 2주 놓치면 모든 잡초가 >3.81cm = "작물" = **영구 사면**. **주말농장은 주말에 간다 — 프로젝트 이름 자체가 반박이다.** 비전이 주는 건 복구 모드다(40일차에 60cm 명아주를 잡초라 부를 수 있음).
3. **오류가 중요도와 역상관이다.** 명아주는 경합강도 0.29(피 0.1707)로 *더 해로운* 잡초이고, 해로운 이유가 **작물을 웃자라 빛을 뺏기 때문**이다. 그 웃자람이 바로 키 규칙에게 "작물"로 보이게 만든다. 규칙의 실패와 잡초의 해악이 **같은 메커니즘**을 공유한다. "분포가 겹친다"보다 훨씬 강하다.
4. **포탈 한 줄 요약(수정)**: "레일 없는 FarmBot"보다 → **"고랑이 곧 레일인 FarmBot — 그래서 남은 문제는 보는 것뿐이다."** 30cm 고랑이 25cm 벽 사이에 있어 **소프트 레일**이 되고 X 위치추정이 기하학에서 거의 공짜로 나온다.

## 2b. 제거 방식 · 구역 분할 · 작물 세트 (사용자 질문 4개에 대한 답)

### 제거 = **스탬핑**. 자르기가 아니라.

**Bosch BoniRob 방식**: 1cm 로드로 잡초를 흙 속 **3cm 깊이로 압입**, 사이클 <100ms, 당근밭 ~90%, 큰 잡초는 반복 타격. **스탬핑은 강체 물리라 Fortress에서 정직하게 시뮬된다** — Z 프리즘 관절 + contact 센서면 끝. 자르기는 연성체가 필요한데 **직접 확인: `libignition-physics5-{bullet,dartsim,tpe}` 전부 softbody 심볼 0개, sdformat12에 deformable 요소 없음, gz-physics#222가 2021-03부터 OPEN + "help wanted"**. 노션의 "자르기 범위 밖"은 결론은 맞고 이유가 약했다(시뮬레이터 핑계는 "더 좋은 시뮬 쓰지"를 부른다). **스탬핑으로 바꾸면 "잡초제거"가 정직하게 범위 안에 들어온다.**

성공 기준을 **kill event**로 승격: `‖tool_xy − weed_gt_xy‖ < 2cm` **AND** dwell **AND** 작물 무접촉. 맨 pose 기준은 선택성 조항이 빠져 있어 부족하다 — 실제로 하고 싶은 주장은 *"…그리고 상추는 안 건드렸다"*이다.

**정정 2개** (내 앞선 주장이 틀림):
- Tertill 바퀴 기여 "16~22%"는 **거꾸로 읽은 것**. 논문 결론 원문: *"Without the string trimmer, Tertill was ineffective"* — 줄날 제거 시 60~72%→4~39%. **바퀴만으론 제초법이 아니다.**
- **FarmBot은 v1.7에서 수동 spike weeder를 삭제**하고 전동 로터리 툴로 교체했다. 가장 가까운 선례가 수동 스탬퍼를 버렸다는 뜻 — 참고할 것.

**논지 확장**: 모든 제초 기술이 **작고 어린 잡초에만** 듣는다(BoniRob·FarmBot·레이저 ED50이 떡잎→4엽에 ~1000배 상승·화염 전부). *"큰 잡초용 제초 기술은 없다. 일찍 제초하는 것만 있다."* 그리고 핑거위더·토션위더 등 고전 기계 제초 도구도 전부 *"size differential between crop and weed"*로 선택 — **핑거위더는 기계로 만든 키 규칙**. → **Tertill만이 아니라 기계 제초 정전 전체가 크기 차이로 선택성을 산다. 카메라가 그 의존을 끊는 것.**

### 고랑 vs 두둑 — 빼는 게 아니라 **대조군으로 쓴다**

한국 농법이 이미 두 구역을 다른 도구로 다룬다(민들레, 원문): *"관행농은 두둑에 비닐 멀칭을 하고 고랑에 제초제를 친다. 유기농은 두둑에 비닐 멀칭을, 고랑엔 부직포를 깔아 풀을 제어한다."* → **고랑=비선택적, 두둑=선택적.** 농부들이 독립적으로 수렴한 분할이고, **선택성이 인식이 사주는 것**이다. 고랑 = **위치추정 문제, 분류 0**(RDA 로봇은 RTK ±10cm로 위치를 사고 분류는 안 함 — 위치추정은 구매 가능한 상품, 분류는 아님).

- 면적: 두둑 90/고랑 30 → **두둑 75%**, 작물 100%, 어려움 전부. "고랑이 더 넓다" 걱정은 사실무근.
- **포탈은 고랑 제초 불가**: 캐리지 ±45cm = 정확히 두둑 90cm, 다리가 ±60cm라 막힘. 바퀴는 고랑 30cm 중 8cm(27%)만 덮음. → 고랑엔 **바퀴 뒤 상시 견인 도구**(인식 0). 한국 농법과 정확히 대응. 자르기가 범위 밖이므로 **아예 안 만든다** — 그게 바로 논거다(기계적 디테일이지 지능이 아님).
- **README 프레이밍**: "고랑은 제초할 필요 없다"고 하면 안 된다(남양주가 *"텃밭 인근 통로 제초"*를 참여자 의무로 명시). 올바른 문장은 **"고랑 잡초는 공짜 점수라서 세지 않는다"** — 포함하면 recall이 부풀 뿐이다. 유리한 카테고리를 뺐으니 숨겼다는 비난이 불가능하다.

**🎯 최고의 실험 — 두 구역을 다 보고하고 대조한다:**

| 구역 | "초록=잡초" HSV 6줄 | 키 규칙(Tertill) | 비전 |
|---|---|---|---|
| **고랑** | ~100%, 작물 피해 0 | ~100% | ~100% (낭비) |
| **두둑** | recall 높음, **작물 몰살** | 키 겹침으로 실패 | **핵심** |

*"우리는 두 구역을 다 보고한다. 고랑은 대조군이다 — HSV 6줄이면 풀린다. 두둑이 과제다 — 같은 임계값이 작물을 몰살한다. 그 격차가 이 로봇에 카메라가 달린 이유다."* 추가로 모든 잡초에 **작물까지의 거리**를 태깅해 거리별 recall 곡선을 그린다. 키 규칙이 죽는 지점이 거기고, 논지가 수사가 아니라 경험적 사실이 된다.

### 🔄 비닐멀칭 위험 — 유리하게 뒤집힘 (프레이밍의 최대 위협이었음)

한국 밭농사는 두둑에 비닐을 깐다. 사실이면 논지가 죽는다. **그런데 타깃 세그먼트에서는 규정으로 금지돼 있다:**
- **서울시 동행서울 친환경농장** (원문): *"유기합성농약, 화학비료를 사용하지 않는 유기농법과 **비닐 멀칭을 하지 않는** 환경친화적인 농법을 준수"*. 잔류농약 검사로 강제, 검출 시 즉시 퇴출. 16.5㎡/구좌.
- **남양주 도시텃밭** (원문): *"**합성농약/ 비닐멀칭을 하지 않는** 친환경 농업 실천"*. 경기도 농업기술센터 동일.

**→ 주말농장은 규정으로 두둑이 맨흙이다.** 그리고 이게 위협이 아니라 **프로젝트를 설명한다**:

> **상업 밭엔 제초 문제가 없다 — 비닐과 제초제가 있으니까. 주말농장에 제초 문제가 있는 건 그 둘이 금지돼서다. 제초 로봇은 비닐+제초제의 여집합이고, 지구상 모든 제초 로봇은 맨흙을 전제한다. 이 세그먼트는 두 산업적 정답의 부재로 정의된다.**

규모(농식품부 2024): 도시농업 참여자 **150.4만명**, 도시텃밭 **952ha**, 2011 도시농업법 이후 ~9.8배 성장.
단, README에 **무멀칭/친환경 세그먼트임을 명시**할 것 — 안 쓰면 리뷰어가 "진짜 경쟁자는 ₩10,000짜리 비닐 한 롤"이라고 지적한다.

### 키 큰 작물 — 배제가 아니라 **"언제 제초하나"의 문제**

내가 **수확기 키를 캡했는데, 중요한 건 제초 시점의 키**였다.

| 고추 | 초장 | 필요 클리어런스 |
|---|---|---|
| 정식 20일 | **18 cm** (1차 검증) | 53 cm |
| 정식 42일 = **경합한계기 끝** | 40~63 cm (출처 2개 불일치, 범위로 보고) | 75~98 cm |
| 수확기 | 194.7 cm | 농학적으로 **무의미** |

**잡초 경합 한계기 = 정식 후 2~6주**(Pyon et al. 1999, Min & So 2016 경유). *"정식 후 6주 이상 제초한 경우 고추 수량이 완전 제초와 비슷"* → 6주 넘겨 제초하면 얻는 게 0. **100cm 포탈이면 고추 전 임계기간 커버.** 150cm 고추는 이미 잡초를 신경 안 쓰는 고추다.

**작물 키 캡은 농학이 아니라 도구 높이 캡이었다 — ecoRobotix 스펙표가 자백:** 같은 표의 인접 두 줄이 *"노즐 램프 15~30cm 높이 조절"* / *"작물 키 30cm까지 제초 가능"*. **같은 숫자.** ARA에서 40cm로 오른 것도 농학이 아니라 도구가 바뀌어서.

**→ 포탈을 높이지 말고 카메라에 Z축을 달아라.** 포탈 높이와 작업거리가 분리돼 트레이드오프가 사라진다. px/cm 산수(HFOV 60°, 640×480): 60cm→**15.84**, 100cm→7.39, 185cm→3.46. **640@60cm가 1280@185cm를 2.3배 이긴다 — 센서로는 높이를 살 수 없다.** 나머지 걱정(빔 처짐 0.063mm, 전복 1.5g)은 시뮬 전용 프로젝트에서 전부 가짜다. px/cm만 진짜다(시뮬이 정확히 재현하니까). 업계도 그렇게 한다(ecoRobotix 램프, FarmBot Z).

### 작물/잡초 세트 — ROC가 저절로 써진다

**주력 = 상추.** 검증된 한국 잡초 초장 대비:

| 잡초 | 초장 | 상추(20~25cm) 대비 |
|---|---|---|
| 쇠비름 *Portulaca oleracea* | **15~30 cm** (포복형) | 50% 더 큼 |
| 닭의장풀 | 15~50 cm | 78.7% |
| 개비름 | 25~80 cm | 100% |
| 바랭이 | 40~70 cm | 100% |
| 흰명아주 | 60~150 cm | 100% |

**한국 잡초의 85.7%가 상추보다 크다. "키 크면 작물" 규칙의 AUC = 0.144 — 약한 게 아니라 역전이다.** 동전 던지기보다 나쁘다.

**논지가 "분포가 겹친다"보다 강해진다**: 명아주는 *더 해로운* 잡초다(경합강도 0.29 vs 피 0.1707, 경제적 피해 한계밀도 3.7 vs 6.5 개체/100㎡). **해로운 이유가 작물을 웃자라 빛을 뺏기 때문이고, 웃자람이 바로 키 규칙이 "작물"이라 부르게 만드는 것이다.** 키 규칙의 오류는 랜덤 노이즈가 아니라 **잡초의 중요도와 역상관**이며, 같은 메커니즘을 공유한다.

**그리고 프로젝트 이름 자체가 반박이다**: Tertill의 2.54/3.81cm 임계값은 **매일 순찰**로 떡잎을 죽여야만 성립한다. **키 규칙은 분류기가 아니라 복구 모드 없는 래칫**이다 — 2주 놓치면 모든 잡초가 >3.81cm = "작물" = **영구 사면**. **주말농장은 주말에 간다.** Tertill의 전제가 이 유스케이스와 구조적으로 양립 불가다.

**정정: CropCraft는 잡초로 `portulaca`, `polygonum`, `taraxacum`을 실제로 배포한다**(내 앞선 브리핑이 틀림 — bean/maize만이라는 건 *작물* 한정). **`portulaca` = 쇠비름 = AI Hub 527과 정확히 같은 종.** 그리고 `max_height`가 YAML 1급 파라미터 → **ROC 실험이 YAML 스윕이고 정답은 공짜.**

**⭐ 쇠비름을 대조군으로 반드시 유지**: 포복형 15~30cm라 **키 규칙이 통하는** 잡초다. 모든 잡초가 규칙을 깨면 리뷰어가 조작이라 한다. 쇠비름이 정직한 음성 사례이고, 이게 ROC를 *허수아비*가 아니라 *결과*로 만든다. 주인공은 `polygonum`(직립형, 규칙을 깸).

**함정 회피**: 상추와 명아주가 둘 다 리스케일된 콩이면 유일한 신호가 **크기**뿐이라 키 분류기를 구조적으로 만들어버리고 "비전이 키를 이긴다"는 주장이 순환논법이 된다. **서로 다른 메시**(bean vs portulaca vs polygonum)가 형태 신호를 준다. 이 성질을 의도적으로 지킬 것.

## 3. 베이스라인 — "로봇을 다 만들어야 하나?"에 대한 답

**둘 다 아니다. 로봇 1대에 결정 노드만 갈아끼운다.** P0/P1/P2는 로봇이 아니라 함수 하나다(각 ~하루).

**발표된 숫자와 직접 비교는 불가능하다 — 이유가 예상과 다름:** 당신은 "잡초 위에 위치"를 재고, 발표 숫자는 전부 "잡초를 **죽인** 결과"(밀도/피복률/생체량)를 잰다. **단위 변환이 없다.** 게다가 Sanchez & Gallandt 2021에 따르면 **Tertill 효능의 16~22%는 바퀴가 밟아 뭉갠 것** — 시뮬하지도, 하려고 하지도 않는 물리다. (일반 원칙의 인용처: BARN 벤치마크 arXiv:2008.13315 — *"there is no generally accepted metric by which to compare different approaches against one another"*. 이 벤치마크가 존재하는 이유 자체가 논문 간 숫자 비교 불가라서다.)

**발표 숫자의 진짜 쓸모** (비교 대상이 아니라):
1. **베이스라인 파라미터의 출처.** P0 임계값은 내가 고르는 게 아니라 **특허가 정한다**(2.54cm 잡초 센서 / 3.81cm 작물 센서). P1 논리는 FarmBot 문서가 정한다. ← BARN 프로토콜 그대로.
2. **허수아비가 아니라는 증거.**
3. **README의 *맥락* 섹션** — 절대 결과 표에 섞지 않는다.

**실험 설계** — 같은 몸체, 같은 시드 정원 N개, 정책만 교체:

| 정책 | 내용 | 비용 |
|---|---|---|
| **P0** Tertill류 | 특허 수치 그대로 키 규칙 + 랜덤워크. 위치 모름 | ~1일 |
| **P1** 줄 기하학 | 줄 위=작물, 줄 밖=잡초. 위치 필요 | ~1일 |
| **P2** FarmBot류 | 심은 좌표 DB + 값싼 HSV 인식(카메라 있음!) | ~1일 |
| **P3** weedwatch | 의미분할 + 커버리지 + 접근 | 본체 |
| **P4** Oracle | 완벽 인식(GT 좌표 직독) = **천장**. 갭이 인식 탓인지 주행 탓인지 분해 | ~2시간 |

**지표**(전부 헤드리스 계산 가능): 표적 잡초 recall · **작물 오검출률**(정원사가 실제로 신경 쓰는 것, Fortress contact 센서/GT로 무료) · 커버리지% · 주행거리 · 시간 · 에너지.

**정직성 장치**: P0 임계값 스윕 ROC 보고(고정 임계값 테스트 금지). 키 분포는 **농학적으로**(모종 이식 높이 vs 실제 잡초 높이) 정하지 적대적으로 정하지 않음. **키 규칙이 이기는 정원을 일부러 포함**(모종기 작물이 주변 잡초보다 작을 때) — 전부 초록불인 표는 아무도 안 믿는다.

## 5. 검증 아키텍처 (하드 제약)

**4단 피라미드:**
- **Tier 0 — 위생.** `scripts/env.sh`: `unset PYTHONPATH AMENT_PREFIX_PATH …; source /opt/ros/humble/setup.bash; source install/setup.bash; exec "$@"`. 여기 안 거치는 명령 금지(CI에서 grep으로 강제).
- **Tier 1 — 순수 단위.** 시뮬·GPU 없음. 미션 FSM을 `(state,event)→(state,action)` 순수 함수로 두고 ROS I/O를 밖으로 빼면 pytest가 밀리초 단위로 전 미션 로직을 검증. **이 구조적 선택 하나가 나머지 전부보다 검증에 기여한다.**
- **Tier 2 — 렌더 없는 시뮬.** `ign gazebo -s -r --iterations N`, 카메라 끔. 주행·제어·수렴. GPU 불필요 → CI 가능.
- **Tier 3 — 렌더 있는 시뮬.** GPU+EGL 필요, 로컬 전용(`@pytest.mark.gpu`). YOLO 인루프.

**키스톤**: `ros_gz_interfaces/srv/ControlWorld`(`pause/step/multi_step/reset/seed`)를 브리지해 pytest가 시뮬을 정확한 스텝 수만큼 전진 → 벽시계 경합(가장 큰 flake 원인) 제거. `use_sim_time:=true` 전 노드.

**단언은 허용오차 기반.** Fortress 비트 결정성은 미확인이고 렌더는 별 스레드다. `dist < 0.05` / `success_rate >= 0.8 over N=20 seeds`로 쓰고, 부동소수 동일성을 요구하는 테스트는 설계 오류로 취급.

**오라클 = `description.json` + `/world/<w>/pose/info`.** segmentation 카메라 아님(Fortress의 인스턴스 분리 입도는 **top-level 모델**뿐이라 CropCraft의 단일 모델 구조에선 잡초 전체가 한 덩어리). 성공 기준이 진술된 단위(미터)를 직접 준다.

**자기기만 방지** (에이전트가 훈련도 평가도 하므로):
- `configs/train_seeds.txt` / `configs/eval_seeds.txt` — **코드가 아니라 데이터로 git에 커밋**, 서로소를 Tier 1 테스트로 단언.

## 6. 스테이지 계획

각 스테이지의 DONE은 **에이전트가 단언할 수 있는 것**이다. 노션의 순서 규율("작동하는 최소 버전 먼저, 대시보드는 마지막")을 지키되 스코프는 전부 포함.

### Stage 0 — 스모크 (첫 커밋. 로봇 코드 이전)
`git init` → `scripts/env.sh` → 카메라 하나 있는 최소 SDF → `ign gazebo -s -r --headless-rendering --iterations N` → PNG 저장 → **검지 않음 AND `GL_VENDOR = NVIDIA`** 단언.
**DONE**: `make smoke` exit 0 + 두 게이트 통과. 이게 통과하면 "에이전트가 스스로 검증한다"는 전제가 하드웨어에서 증명된다. 실패하면 6주차가 아니라 1일차에 안다.
동시에: `python3 -c 'import rclpy'`가 3.10에서 성공 단언, `apt install ros-humble-ros-gz`, Fortress 3종 업그레이드(sensors 6.9.0/rendering 6.6.4/gazebo 6.18.0).

### Stage 1 — 정원과 오라클
Blender 5.0.1 설치 + GPU userpref 프로비저닝 → CropCraft 설치 → **두둑/고랑 ridge 메시 직접 제작**(CropCraft에 지형 개념이 없음 — 이게 반나절 리스크 실험이자 포탈 설계의 전제) → 주말농장 YAML(1종=1bed, `bed_width:0`, `shift_next_bed:false`, y_function 해시 스캐터, 종별 비공약수 `plant_distance`, per-bed `offset:[0,0,0.25]`로 두둑 위에 얹기) → **머티리얼 변환 패스**(검은 렌더 수정) → **채점 잡초를 `beds:`에 선언** → `description.json`에서 잡초 GT 좌표 추출.
**DONE**: 시드 고정 재생성이 동일 `description.json`을 내고, Gazebo 카메라 프레임이 검지 않고 초록이 보이며, 잡초 GT 좌표 리스트가 나오고, 두둑 위/고랑에 잡초가 각각 존재한다(구역 대조 실험의 전제).

### Stage 2 — 로봇 몸체
build123d → STEP/STL → `f3d --backend=egl` 헤드리스 렌더 → **Claude가 PNG를 Read하고 고침** → Blender로 PBR+OBJ export → URDF/xacro. 시각 메시(OBJ)와 충돌 프리미티브 분리. Fortress 내장 `diff-drive`(ros2_control은 나중 리팩터).
**관절 3개**: 베이스 X(주행) + 캐리지 **Y** 프리즘(±45cm, 두둑 폭) + **Z** 프리즘(카메라·스탬퍼 하강). Z가 핵심 — 포탈 높이와 작업거리를 분리해 고추 v2를 연다. 둘 다 `joint-position-controller`(설치 확인됨). 하방 카메라는 캐리지에 강체 고정 → **툴 팁이 고정 픽셀에 옴** → `‖weed_px − tool_px‖ < N`이 헤드리스 단언이 된다.
**DONE**: `ign sdf -p`에 `<pbr>` 존재, `--inertial-stats` 통과, cmd_vel로 주행, Y/Z 관절이 목표 위치 도달, 하방 카메라가 지면 프레임 발행.

### Stage 3 — 인식
AI Hub 527 쇠비름 3GB 선택 다운로드. CropCraft render(1,050장이면 충분 — 논문 헤드라인이 1만이 아니라 1,050장이다) → 의미분할 학습 → **held-out 시드 정원에서 평가** → **한국 실제 쇠비름 사진에서 평가**(단일 클래스 전이, 정직하게 스코프 명시).
**DONE**: eval 시드에서 mIoU 보고, 한국 실사진 전이 수치 보고. 둘 다 `make eval`로 재현.

### Stage 4 — 주행 + 정밀 위치 (**핵심 산출물**)
Nav2에서 AMCL·map_server 뺀 구성(RPP 컨트롤러, MPPI 아님 — 크리틱 30개는 저녁을 잡아먹는다). 보스트로페돈 ~50줄(opennav_coverage는 Humble에 **없음**, 게다가 5×5m엔 과잉). 위치추정 단계별 라벨(**L0 GT=치팅** → L1 휠오도메트리 → L2 +IMU/EKF → **L3 AprilTag 코너 말뚝 = Tertill의 물리적 울타리를 시각적 울타리로 대체 = 온논지**). "앞에서 보고 뒤에서 실행" — 시각 서보 안 함(업계 표준이 그렇고, 마지막 20cm는 데드레커닝 1~2mm).
**DONE (kill event)**: `‖tool_xy − weed_gt_xy‖ < 2cm` **AND** dwell 충족 **AND** 작물 무접촉이 20개 시드에서 ≥80%. 스탬퍼는 Z 하강 + contact 센서 — 강체라 정직하게 시뮬됨.

### Stage 5 — 실험 (**결과물**)
P0/P1/P2/P4 정책 + 시드 정원 N개 → 표. P0 임계값 스윕 ROC(고정 임계값 테스트 금지).
**추가 — 구역 대조**: 고랑/두둑 각각에 대해 HSV 6줄 · 키 규칙 · 비전을 돌려 2×3 표. 고랑에선 셋 다 ~100%, 두둑에선 HSV가 작물을 몰살. **그 격차가 논지의 정량화.** 잡초별 **작물까지 거리** 태깅 → 거리별 recall 곡선.
**DONE**: `make experiment`가 mean±std 표 + ROC + 구역 대조표 + 거리-recall 곡선을 낸다.

### Stage 6 — 배터리 · 도킹
`LinearBatteryPlugin`은 이미 설치돼 있고 예제 월드가 `/usr/share/ignition/ignition-gazebo6/worlds/linear_battery_demo.sdf`에 있음(15줄, 거의 공짜). 배터리는 **일찍** 넣어도 됨(도킹과 분리). `ros-humble-opennav-docking` 0.0.2가 **Humble apt에 존재**(백포트 불필요 — 이 스레드의 전제가 반대로 틀렸음). 도킹이 늦은 이유는 패키지가 아니라 **위치추정 선행조건**.
**DONE**: 20% 배터리에서 도크 복귀 → 충전 → 중단 지점부터 커버리지 재개.

### Stage 7 — 관찰 · 대시보드 (마지막, 노션 규율)
SQLite + 디스크 JPEG(에이전트가 `sqlite3 db "SELECT COUNT(*)"`로 단언 가능. Grafana 패널엔 못 함). rosbag2 `-s mcap`(Humble 기본은 sqlite3 — 명시 필요) → Foxglove. `report.py` → 정적 HTML.
**성장 시계열 트릭**: CropCraft 식물이 높이 bin으로 나뉘고(콩 0.05/0.135/0.217m) 각 변종이 GT `leaf_area`를 가짐 → 같은 시드로 3단계 생성 = "주 1회 방문". **3단계까지만 정직**(그 사이는 메시 리스케일). **익음 판정은 완전 삭제** — 에셋에 열매 지오메트리가 아예 없어 "어려운" 게 아니라 **불가능**.

## 8. 검증 방법 (이 계획이 맞는지 확인하는 법)

```bash
make smoke        # Stage 0: 헤드리스 EGL + NVIDIA 게이트 2개
make doctor       # python 3.10 + rclpy + torch import 단언
colcon test && colcon test-result --verbose --all
make eval         # 인식: held-out 시드 + 한국 실사진 전이
make experiment   # 정책 표 + ROC
```

**가장 먼저 무너질 것 3개, 각각 하루 안에 노출되는 테스트:**
1. **헤드리스 EGL이 인텔/llvmpipe를 잡음** — 이미 로그로 확인됨 → Stage 0의 두 번째 게이트가 잡는다.
2. **CropCraft가 Fortress에서 검게 렌더** — Stage 1 첫 프레임에서 잡힌다.
3. **CropCraft가 두둑/고랑을 못 만듦 — 이제 확정.** 독립 검증 2회로 확인: 지형/고도/heightmap 파라미터가 아예 없고, CropCraft의 "bed"는 **논리적 줄 그룹**(`bed_width`, `rows_count`, `offset`)이지 융기된 두둑이 아니다. **CropCraft는 식물 전용이고, 두둑/고랑 지오메트리는 직접 만들어야 한다**(커스텀 SDF ground 메시). 포탈 설계 전체가 여기 걸림 → Stage 1에서 반나절 실험. 실패 시 클리어런스 60→35cm로 줄고 형태 결정이 흔들린다.

## 10. 신뢰도 경계 (README에 쓰기 전 재확인 필요)

리서치가 스스로 표시한 것들 — 인용 전 1차 확인:
- **고추 초장만 1차 검증됨.** 상추(20~25) · 배추(30~40) · 감자(50~60) · 토마토(100~200)는 **작업 추정치**. RDA는 재식거리와 수확시기는 내지만 초장 표를 안 낸다. RDA 출처로 인용 금지.
- **상추 재식거리 20cm의 출처 미확정.** 충북 농업기술원은 잎상추 25 / 결구상추 30cm라 함. **다행히 Tertill 논거는 20~30cm 전 범위에서 살아남는다**(매뉴얼이 ≥30.5cm를 요구하므로). 범위로 쓰는 게 오히려 강하다.
- **Pyon et al. 1999(경합한계기 2~6주)는 Min & So 2016 경유 재인용.** 원문 미확보 — 1차에서 한 다리 건너.
- **AUC 0.144는 성숙기 잡초 키 기준** = "잡초가 놓쳐진" 시나리오. Tertill 설계대로(매일 순찰, 2.54cm 미만 떡잎 제거)의 공정한 테스트가 아니다. **두 체제를 다 서술할 것** — 안 그러면 리뷰어가 잡는다. 정직한 프레이밍은 "규칙이 분류를 못 한다"가 아니라 **래칫/복구불가** 논거다.
- **범위 내 균등분포 가정은 데이터가 아니라 내 가정.** 실제 키 분포는 우편향. AUC의 *방향*(역전)은 robust하지만 0.144라는 값은 아니다.
- **명아주가 7일 만에 3.81cm를 넘는다는 증거 없음.** 잡초 생장속도(cm/day) 데이터를 못 찾음 → 래칫 논거는 **구조적이고 미정량**. 글에 넣기 전 표적 검색 1회 필요.
- **CropCraft 잡초 목록(portulaca/polygonum/taraxacum)은 설정 문서의 *예시*로 표기됨** — 작물처럼 "available types" 확정 목록이 아니다. clone 후 assets 디렉토리 실물 확인.
- **AVO(30cm)→ARA(40cm) "도구가 바뀌니 캡이 움직였다"는 시사적이지 증명이 아님**(다른 제품). 강한 증거는 AVO 스펙표의 인접 두 줄(램프 15~30 / 캡 30)이다.
- **plot 크기 vs 로봇 크기 긴장 미해소**: 서울 16.5㎡ · 남양주 16㎡ ≈ 5평 = 두둑 1~2개. **128cm 폭 30kg 포탈은 5평 밭에 큰 기계다.** 민간 주말농장 10~30평의 보편성은 미확인. 데모 정원 크기를 의도적으로 정할 것.
