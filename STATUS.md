## ✅ Stage 0 — 스모크 (완료)

**질문**: 사람이 안 보고 있어도 시뮬을 돌려서 진짜 GPU 사진을 받아올 수 있는가?
이게 안 되면 "Claude가 스스로 검증한다"는 프로젝트의 대전제가 무너진다.

**답: 된다.** `make smoke` 통과.

```
사진 장수   : 121
평균/표준편차: 142.00 / 36.12
GL_VENDOR   : NVIDIA Corporation
GL_RENDERER : NVIDIA GeForce RTX 4060 Laptop GPU/PCIe/SSE2
```

가는 길에 함정 3개를 밟았고 전부 `CLAUDE.md`에 기록했다:

1. **EGL이 인텔 내장 그래픽을 잡고 있었다.** RTX 4060이 EGL 장치 목록에 아예 없었다.
   NVIDIA ICD를 고정해서 해결. → 그래서 모든 시뮬 테스트는 게이트가 2개다
   (사진이 나왔나 **AND** NVIDIA가 그렸나). 사진만 보면 거짓 통과한다.
2. **카메라는 구독자가 있을 때만 렌더링한다.** 구독자 0 → 사진 0장, 에러 메시지 없음.
   `tools/run_headless.sh`가 구독자를 붙여준다.
3. **`pkill -f 'ign gazebo'`는 자기 자신을 죽인다.** 자기 명령줄에도 그 문자열이 있어서.
   `'[i]gn gazebo'`로 해결.

## ✅ Stage 1-a — 두둑/고랑 기하학 (완료, 위험 제거됨)

**이번 단계 최대 위험이었다**: CropCraft에 지형 개념이 아예 없어서(바닥이 평면)
두둑을 직접 만들어야 하는데, 포탈형 설계 전체가 두둑 높이 25cm에 걸려 있었다.

**가장 싸게 죽였다**: 위험의 본질은 "로봇이 탈 수 있는 기하학인가"이지
"CropCraft와 통합되나"가 아니다. 그래서 **Blender 없이 상자 프리미티브만으로** 만들었다.
반나절 예상 → 30분.

**결과: 된다.** `make test` 21개 통과 · `make garden` 렌더 확인.

```
두둑 2개 · 폭 90 · 높이 25 · 고랑 30cm
→ 로봇 트랙 120cm · 전폭 128cm · 바퀴 좌우 여유 11.0cm/쪽
```

**여기서 배운 것 — 눈으로 본 게 버그를 잡았다.** 첫 렌더에서 로봇이 두둑이 아니라
**고랑을 걸터타고 있었다** — 바퀴가 두둑 위에 올라가고 몸통이 고랑 위에 뜬, 설계와
정확히 반대인 모습이었다. 두둑이 짝수 개면 y=0이 고랑 한가운데인데 로봇을 거기 세워서다.
숫자(트랙 120, 여유 11cm)는 전부 "통과"했고 사진을 보고서야 알았다.
→ 그래서 `tests/test_garden_geometry.py`가 이제 산수로 검사한다. 눈은 확장이 안 된다.

**같이 확정된 것**: v1 작물은 **상추만**. 배추(40cm)는 75cm 포탈이면 들어오지만 안 올린다 —
CropCraft가 작물로 콩/옥수수만 배포해서 상추도 배추도 결국 같은 콩 메시를 크기만 바꾼 것이
되고, 그러면 둘을 가르는 신호가 크기뿐이라 "비전이 키 규칙을 이긴다"가 순환논법이 된다.
형태 다양성은 잡초 쪽(portulaca/polygonum/taraxacum — 서로 다른 메시)에서 온다.

**남은 v1 단순화** (나중에 갚을 빚): 두둑이 옆면 수직인 상자다. 진짜 두둑은 비스듬하다.
"예쁜가"가 아니라 "탈 수 있나"를 물었으므로 지금은 충분하다.

## ✅ Stage 1-b — Blender + Cycles GPU (완료)

Blender **5.0.1** (snap `5.0/stable` 고정, 번들 파이썬 3.11.13).

**함정 확인 + 해결**: 새로 깐 Blender는 `compute_device_type = 'NONE'`이라
Cycles가 **경고 한 줄 없이 CPU로 렌더링**한다. 스크립트에서 `device='GPU'`라고 써도
조용히 무시된다. 환경변수 우회 없음 — 설정 파일을 만드는 게 유일한 방법.
→ `make blender-gpu` (OPTIX로 켜고 `userpref.blend` 저장). `make doctor`가 매번 검사한다.

**그 검사가 진짜 실패하는지도 확인했다**: 설정 파일을 치우면 `exit=2`로 죽으면서
원인과 고치는 법을 출력한다. 통과만 확인하면 절반이다.

**측정된 렌더 시간** (1920×1080 / 32샘플 / 식물 800개 / OPTIX):

```
장면 빌드   8.06s   (여러 장에 분산됨)
장당 렌더  11.2s   (11.60 → 11.14 로 수렴)
```

→ **논문 헤드라인 재현(1,050장) ≈ 3.5시간.** 하룻밤이면 된다.
리서치 추정(5~10초/장)보다 느리지만 결론은 유지 — 일주일이 아니다.
그리고 논문 헤드라인이 1만 장이 아니라 **1,050장**이었다는 게 여기서 값을 한다.
(공개된 1만 장 데이터셋 `doi:10.57745/TNCSLP` 도 있어서 첫 베이스라인엔 렌더링이
아예 불필요할 수도 있다.)
